* 코드 정리 구분
    * PR은
        * 기능 변경과 정리를 구분하고
        * 가능한 작은 단위로 커밋하고 생성하는 것이 좋다
    * 위 방법의 이점
        * 테스트
            * 코드 정리는 동작이 바뀌지 않아야 하므로 정리 후 테스트를 통과했다면 동작이 변하지 않았다는 증거
        * 리뷰가 용이함
        * 버그 추적 쉬움


* 연쇄적인 정리
    * 코드를 정리하다보면 그 정리를 완성하기 위해 연쇄적으로 정리가 필요해져서 수정 코드가 점차 거대해지는 상황이 발생함
    * 이 경우에는 무작정 확장하지 말고 멈추는 규칙과 관리 방식을 가져가야함
        * 변수명 하나 바꾸려다 → 관련 함수명도 어색
        * 함수 하나 추출하려다 → 중복이 보여서 제거하고 싶음
        * 중복 제거하려다 → 호출 구조가 헷갈려서 재배치 필요
        * 재배치 하려다 → 파일 구조/모듈 경계까지 건드리고 싶음
    * 정리 코드 범위가 커지면
        * 리뷰가 어려움
        * 테스트/검증 비용 급증
    * 멈추는 방법 예시
        * 연쇄가 발생하더라도 그 작업을 작은 단위로 쪼개어 정리한다.
        * 정리의 목표를 미리 정해두고 해당 작업에 연관된 정리만 진행한다.


* 코드 정리의 일괄 처리량
    * 통합/배포 시 적절한 코드 정리 크기는 어떻게 정해야할까?
    * 코드 정리시 고려해야할 비용
        * 충돌
            * 일괄 처리하는 코드가 많을수록 통합 과정에서 지연시간이 길어지고, 협업자의 코드와 충돌할 가능성이 높아짐
        * 상호작용
            * 다수의 코드 정리를 처리하다가 우연히 동작 변경을 할 수도 있음
        * 추측
            * 코드 정리 범위가 커질수록 연쇄적으로 정리하는 코드가 발생되고, 예상하지 못한 작업이 계속 발생한다.
    * 이런 검토 비용을 제대로 줄일 수 있는 방법을 찾아야함


* 리듬
  * 코드 정리는 몇분에서 한시간을 넘지 않도록 한다 
  * 한시간을 넘어가는 정리 작업은 나누어서 한다
  

* 얽힘 풀기 
  * 코드 정리를 마치고 기능 변경중 추가적으로 코드 정리가 필요해질 경우 
  * 기존 코드를 버리고 정리를 선행한 후 다시 기능을 변경하는 것을 권장한다.


* 코드 정리 시점 
  * 정리 하지 않는다.
    * 기능이 절대 변경되지 않는다면(그럴일이 거의 없음)
  * 나중에 정리하기 
    * 정리가 필요한 코드 목록을 정리해 두었다가, 시간이 날때 할애 가능한 시간내에 작업이 가능한 항목을 먼저 처리해보자 
  * 동작변경 후 정리
    * 방금 고친 코드를 다시 변경할 예정이거나
    * 곧 바로 정리할것일때
    * 코드 정리하는데 드는 시간이 동작 변경에 드는 시간과 거의 비슷할때  
  * 코드 정리후 동작변경 
    * 이제 제일 권장되는 방식이지만, 무겁게 생각하지는 말자


* 코드를 정리해야하는 이유 
  * 소프트웨어 설계; 요소들을 유익하게 관계맺는 일 
  * 요소의 계층 구조(토큰-식-문-함수-객체/모듈-시스템)
  * 요소 사이의 관계 
    * 이러한 관계가 만들어내는 이점


* 구조와 동작 
  * 실제 소프트웨어의 가치는 동작에서 나온다 
  * 하지만 확장 가능한 구조는 선택 가능성을 높여준다
  * 그렇지만 절 설계된 구조가 무조건 소프트웨어의 가치를 높인다고 할 수는 없다.
