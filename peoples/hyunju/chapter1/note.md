* 정리란 무엇인가?
  * 코드의 "동작"은 바꾸지 않고 "구조"만 개선하는 행위
  * 가독성, 이해도, 변경 용이성을 높이는 것이 목적
* 언제 정리 해야 하는가?
  * 기능 변경 중 이해가 막히는 순간
  * 중복, 긴 메서드, 의미 없는 이름을 만났을때
  * 코드가 버려질 예정이라면 굳이 하지 않는다.
* 언제 어떻게 정리 하는가?
  * 한번에 크게 하지 않고 아주 작은 단위로
  * 구조 변경과 동작 변경을 절대 섞지 않는다.
1. 보호 구문 
   1. 중첩된 if 문을 줄여서 가독성 향상
   2. 실패, 예외, 무의미 케이스의 위치를 위로 옮긴다.
   3. 너무 많은 guard가 나열되면 검증을 별도의 함수로 추출한다.
2. 안쓰는 코드
   1. 사용되지 않는 코드는 제거하거나 격리한다.
3. 대칭으로 맞추기
   1. 비슷한 일을 하는 코드들은 비슷한 모양을 가지도록 정리한다.
      1. 예: 모든 함수에서 검증 로직을 상위로 통일한다.
      2. 컬렉션 처리 시 for문을 쓰는 것으로 통일 or stream() 쓰는 것으로 통일
      3. 네이밍 대칭 (생성은 create, 삭제는 delelte, 조회는 get prefix 사용)
4. 새로운 인터페이스로 기존 루틴 부르기
    1. 현재 코드가 너무 복잡하여 바로 고치기에는 부담스러울때, 내가 만들고 싶은 구조의 인터페이스를 만들고, 해당 인터페이스에서 기존 로직을 호출하도록 설계
   2. 호출부에서 복잡한 로직을 숨기고, 이후 리팩터링 시 외부 변경을 최소화 한다.
5. 사람이 코드를 읽는 순서대로 코드를 재배치한다.
   1. 사람은 코드를 위에서 아래로, 추상에서 구체로 읽으므로 사람이 이해하기 쉽도록 코드를 배치하는 것
   2. 코드의 목적 -> 큰 흐름 -> 세부 구현 -> 예외/특수 케이스
   3. 배치와 순서만 바꾸는 작업
6. 응집도를 높이는 배치
   1. 함께 변하는 코드들을 물리적으로 가까이 배치한다.
7. 선언과 초기화를 함께 옮기기
   1. 변수가 의미를 얻는 순간에 선언과 초기화를 함께 둔다.
   2. 선언과 초기화가 멀리 떨어져 있으면 읽는 사람은 불필요한 추적을 하게됨
